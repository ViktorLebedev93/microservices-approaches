## Задача 1: Обеспечить разработку

Предложите решение для обеспечения процесса разработки: хранение исходного кода, непрерывная интеграция и непрерывная поставка.

Решение должно соответствовать следующим требованиям:
- облачная система;
- система контроля версий Git;
- репозиторий на каждый сервис;
- запуск сборки по событию из системы контроля версий;
- запуск сборки по кнопке с указанием параметров;
- возможность привязать настройки к каждой сборке;
- возможность создания шаблонов для различных конфигураций сборок;
- возможность безопасного хранения секретных данных (пароли, ключи доступа);
- несколько конфигураций для сборки из одного репозитория;
- кастомные шаги при сборке;
- собственные докер-образы для сборки проектов;
- возможность развернуть агентов сборки на собственных серверах;
- возможность параллельного запуска нескольких сборок;
- возможность параллельного запуска тестов.

## Решение 1:

# Решение для обеспечения процесса разработки

## Основная платформа: **GitLab SaaS (gitlab.com)**

### Соответствие требованиям:

| Требование | Реализация в GitLab |
|------------|---------------------|
| **Облачная система** | GitLab.com (управляемый SaaS) |
| **Git система контроля версий** | Встроенный Git репозиторий |
| **Репозиторий на каждый сервис** | Отдельный проект на сервис |
| **Запуск сборки по событию из Git** | Pipeline triggers на push/merge/tag |
| **Запуск сборки по кнопке с параметрами** | Manual pipelines с переменными |
| **Настройки к каждой сборке** | CI/CD Variables (per pipeline) |
| **Шаблоны для конфигураций** | `.gitlab-ci.yml` с includes |
| **Безопасное хранение секретов** | Masked/Protected Variables, Vault интеграция |
| **Несколько конфигураций из одного репо** | Multiple jobs/stages с rules |
| **Кастомные шаги при сборке** | Custom scripts в job'ах |
| **Собственные докер-образы** | Docker in Docker, любые образы |
| **Собственные агенты сборки** | Self-hosted GitLab Runners |
| **Параллельный запуск сборок** | Parallel jobs |
| **Параллельный запуск тестов** | Parallel test jobs, test splitting |


## Ключевые преимущества GitLab:

### 1. Единая интегрированная платформа:
- Git репозитории + CI/CD + Container Registry в одном месте
- Уменьшение контекстных переключений и интеграционных сложностей

### 2. Гибкость инфраструктуры:
- GitLab SaaS для управления и хранения кода
- Self-hosted runners для контроля над средой сборки
- Возможность использовать наши Docker образы

### 3. Безопасность и управление секретами:
- Masked и Protected переменные для паролей и ключей
- Интеграция с HashiCorp Vault для продвинутого управления секретами
- Изоляция сборок через Docker контейнеры

### 4. Производительность и масштабирование:
- Параллельное выполнение сборок и тестов
- Распределенные runners для разных типов задач
- Кэширование зависимостей и артефактов

## Альтернативы и почему GitLab лучше:

### GitHub Actions:
- Плюсы: Широкая экосистема, интеграция с GitHub
- Минусы: Разрозненные инструменты, сложнее управление секретами между системами

### Jenkins + GitHub/GitLab:
- Плюсы: Максимальная гибкость, множество плагинов
- Минусы: Сложная настройка и поддержка, нет единой платформы

## Вывод:

GitLab SaaS с self-hosted runners является оптимальным решением для крупной компании с микросервисной архитектурой. Оно полностью соответствует всем требованиям ТЗ, предоставляя облачную систему с полным контролем над инфраструктурой сборки, безопасным хранением секретов и возможностью масштабирования под нужды сотен микросервисов.

## Задача 2: Логи

Предложите решение для обеспечения сбора и анализа логов сервисов в микросервисной архитектуре.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- сбор логов в центральное хранилище со всех хостов, обслуживающих систему;
- минимальные требования к приложениям, сбор логов из stdout;
- гарантированная доставка логов до центрального хранилища;
- обеспечение поиска и фильтрации по записям логов;
- обеспечение пользовательского интерфейса с возможностью предоставления доступа разработчикам для поиска по записям логов;
- возможность дать ссылку на сохранённый поиск по записям логов.

Обоснуйте свой выбор.

## Решение 2:

## Выбранный стек: EFK (Elasticsearch, Fluentd, Kibana)


### Компоненты

#### 1. Fluentd
- **Роль**: Агент сбора логов
- **Развертывание**: DaemonSet в Kubernetes или агент на каждой VM
- **Источники**: stdout/stderr контейнеров, файлы логов
- **Особенности**: Встроенная буферизация, гарантированная доставка, парсинг JSON

#### 2. Elasticsearch
- **Роль**: Хранилище и поисковый движок
- **Конфигурация**: Кластер из 3+ узлов
- **Хранение**: Индексы с ротацией по дням
- **Поиск**: Полнотекстовый, фильтрация по полям

#### 3. Kibana
- **Роль**: Веб-интерфейс для анализа
- **Возможности**:
  - Поиск и фильтрация логов
  - Визуализации и дашборды
  - Сохраненные поиски с shareable ссылками
  - RBAC для разработчиков

### Реализация требований

| Требование | Реализация |
|------------|------------|
| **Сбор со всех хостов** | DaemonSet Fluentd или агенты на ВМ |
| **Минимальные требования** | Логи в stdout, автоматический сбор |
| **Гарантированная доставка** | Буферизация Fluentd + retry механизмы |
| **Поиск и фильтрация** | Elasticsearch + Kibana Query Language |
| **Пользовательский интерфейс** | Kibana с веб-доступом |
| **Ссылки на поиски** | Сохраненные объекты Kibana с прямой ссылкой |

### Преимущества

- **Простота интеграции**: Приложения только пишут в stdout
- **Масштабируемость**: Горизонтальное масштабирование Elasticsearch
- **Надежность**: Гарантированная доставка через буферизацию
- **Экосистема**: Широкая поддержка плагинов и интеграций

### Безопасность

- TLS для передачи между компонентами
- Аутентификация в Kibana (LDAP/SSO)
- Разграничение прав доступа к логам по командам

### Мониторинг стека

- Health checks Elasticsearch
- Мониторинг lag в обработке логов
- Алертинг на сбои компонентов

### Альтернативы

- **Для высоких нагрузок**: Kafka между Fluentd и Elasticsearch
- **Для ограниченных ресурсов**: Loki+Grafana вместо ELK
- **Для облачных сред**: Managed сервисы (Amazon OpenSearch, etc.)

Данное решение является отраслевым стандартом и покрывает все требования по сбору, доставке, хранению и анализу логов в микросервисной архитектуре.


## Задача 3: Мониторинг

Предложите решение для обеспечения сбора и анализа состояния хостов и сервисов в микросервисной архитектуре.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- сбор метрик со всех хостов, обслуживающих систему;
- сбор метрик состояния ресурсов хостов: CPU, RAM, HDD, Network;
- сбор метрик потребляемых ресурсов для каждого сервиса: CPU, RAM, HDD, Network;
- сбор метрик, специфичных для каждого сервиса;
- пользовательский интерфейс с возможностью делать запросы и агрегировать информацию;
- пользовательский интерфейс с возможностью настраивать различные панели для отслеживания состояния системы.

Обоснуйте свой выбор.

## Решение 3:

### Стек: Prometheus + Grafana

### Архитектура


### Компоненты

#### 1. **Prometheus**
- Сбор метрик по pull-модели
- Хранение временных рядов
- Язык запросов PromQL
- Service discovery для автоматического обнаружения сервисов

#### 2. **Node Exporter**
- Сбор метрик хостов: CPU, RAM, Disk, Network
- Запускается на каждом хосте
- Стандартный экспортер для системных метрик

#### 3. **cAdvisor** (для контейнеров)
- Сбор метрик ресурсов контейнеров
- CPU/RAM/Disk/Network на уровне сервисов
- Интеграция с Docker/Kubernetes

#### 4. **Grafana**
- Визуализация и дашборды
- Поддержка PromQL запросов
- Настраиваемые панели и графики
- Алертинг и уведомления

### Реализация требований

| Требование | Реализация |
|------------|------------|
| Сбор метрик со всех хостов | Node Exporter на каждом хосте |
| Метрики ресурсов хостов | Node Exporter (CPU/RAM/Disk/Network) |
| Метрики ресурсов сервисов | cAdvisor + application метрики |
| Специфичные метрики сервисов | Custom exporters / инструментация |
| UI для запросов и агрегации | Grafana + Prometheus UI |
| Настраиваемые панели | Grafana дашборды |

### Сбор специфичных метрик

1. **Инструментация приложений**:
   - Prometheus клиентские библиотеки
   - Экспозиция метрик на `/metrics` endpoint
   - Кастомные метрики бизнес-логики

2. **Экспортеры для внешних сервисов**:
   - PostgreSQL, Redis, Kafka, Nginx
   - Готовые экспортеры из сообщества

### Преимущества

- **Минимальные накладные расходы**: Легковесные экспортеры
- **Автоматическое обнаружение**: Service discovery в динамических окружениях
- **Гибкость запросов**: Мощный язык PromQL
- **Экосистема**: Большое количество готовых решений
- **Kubernetes интеграция**: Нативная поддержка контейнерных сред

### Безопасность

- TLS для трафика метрик
- Аутентификация в Grafana
- Разграничение прав доступа к дашбордам
- Изоляция метрик по командам/проектам

### Масштабирование

- **Вертикальное**: Увеличение ресурсов Prometheus
- **Горизонтальное**: Thanos или Cortex для распределенного хранения
- **Федерация**: Иерархическая структура сбора метрик

Данное решение является отраслевым стандартом для мониторинга микросервисных архитектур и полностью покрывает все требования.
